# KOC YouTube Product Feedback Dashboard PRD

## 1. Background

我们希望借由这个工具，在团队不用完整看完相关 KOC 发布视频的情况下，也能快速知道他们对我们产品说了什么。

当前的问题是：

- KOC 视频内容分散，人工逐条观看成本高。
- 视频里真正和产品相关的反馈常常混在游戏实况、闲聊、剧情讨论或其他内容里。
- 产品团队需要的是可执行的反馈和建议，而不是完整视频转写。
- 如果只给 AI 总结，又容易缺少证据片段，产品同事难以判断反馈是否可信。

因此，第一版 Dashboard 的重点不是做一个复杂的舆情系统，而是做一个能从指定 YouTube KOC 内容中提取产品有效反馈的工作台，并最终输出一版可直接复制给产品团队参考的文字。

## 2. Product Goal

构建一个面向内部团队的 KOC 视频分析 Dashboard，用于：

- 用户创建产品项目，并添加想关注的 YouTube KOC 频道。
- 基于项目中的产品名称和关键词，自动分析对应 KOC 频道下的视频内容。
- 自动提取视频中和目标产品/游戏相关的内容。
- 识别 KOC 对产品的有效反馈、建议、评价和正面/中性/负面倾向。
- 保留原始证据，包括引用片段、时间戳和视频来源。
- 生成一版可以直接复制给产品团队的结构化文字总结。

## 3. Target Users

主要用户：

- 市场团队：希望了解 KOC 内容中对产品的评价和传播效果。
- 发行团队：希望快速判断 KOC 是否提到产品卖点、痛点和用户反应。
- 产品团队：希望接收整理后的有效反馈和改进建议。

次要用户：

- 内容运营：希望判断哪些视频值得进一步人工精看。
- 项目负责人：希望快速看到多个 KOC 对某个产品的反馈概览。

## 4. Core User Stories

1. 作为市场同事，我想创建一个产品项目，并添加想关注的 KOC 频道，让系统自动告诉我这些 KOC 的视频里有哪些内容和我们的游戏有关。
2. 作为发行同事，我想看到 KOC 对我们产品的正向反馈、负向反馈和中性描述，避免完整看完视频。
3. 作为产品同事，我想收到一版已经整理好的文字，里面包含有效反馈、建议和证据片段，方便判断是否需要跟进。
4. 作为项目负责人，我想看到多个视频的汇总结果，快速判断 KOC 反馈里是否出现重复问题或高频建议。

## 5. MVP Scope

第一版覆盖“产品项目 -> 添加指定 KOC 频道 -> 自动获取并分析该 KOC 频道视频”的闭环，不做全网自动监控。

### 5.1 Input

MVP 的必填输入：

- 目标产品/游戏名称。
- 产品相关关键词，例如角色名、玩法、版本名、活动名、竞品名。
- KOC YouTube 频道链接。

系统需要提供基础默认关键词，用户可以在默认关键词基础上增删。默认关键词建议包含：

- `bug`
- `lag`
- `crash`
- `fps`
- `loading`
- `freeze`
- `stutter`
- `paywall`
- `gacha`
- `grind`

MVP 的可选输入：

- KOC 名称或备注。
- 可选：人工补充的产品背景，例如当前版本、关注点或已知争议。

### 5.2 Processing

系统需要完成以下处理：

- 创建或选择产品项目，并保存产品名称和关键词。
- 根据 KOC YouTube 频道链接获取频道基础信息和对应视频列表。
- 自动将该频道下待分析的视频加入分析队列。
- 获取视频基础信息：标题、频道、发布时间、视频链接。
- 获取每个待分析视频的字幕或转写文本。
- 从字幕中筛选和目标产品/游戏相关的片段。
- 对相关片段进行语义聚类，避免同一观点重复出现。
- 识别每条反馈的正面/中性/负面倾向、重要性和证据。
- 生成项目级可复制产品反馈总结。

### 5.3 Output

MVP 输出包含两类结果：

1. Dashboard 内展示的分析结果。
2. 可复制给产品团队的项目级文字总结。

## 6. Analysis Dimensions

系统需要尽量把 KOC 的表达转化成产品团队能理解和使用的反馈。

### 6.1 Feedback Polarity

每条有效内容只需要归类为以下倾向之一，第一版不再单独维护复杂反馈类型：

- Positive：KOC 明确认可、称赞或表达正向态度的内容。
- Neutral：KOC 做出的事实描述、信息转述、轻度观察，暂不体现明确正负态度的内容。
- Negative：KOC 明确批评、不满、质疑，或提到可能影响口碑、传播、体验的问题。

改进建议不作为单独分类，而是在报告生成阶段基于 Positive、Neutral、Negative 内容归纳出来。

### 6.2 Classification Scope

MVP 不做产品领域细分。所有有效内容只归入 Positive、Neutral、Negative 三类。

### 6.3 Sentiment Handling

情绪倾向和 Feedback Polarity 合并，不再单独维护一套情绪字段。

正面/中性/负面判断必须和证据片段绑定，不能只给没有依据的整体判断。

### 6.4 Importance

重要性分为：

- High：可能直接影响产品决策、版本优化、口碑风险或核心卖点判断。
- Medium：有参考价值，但需要更多样本验证。
- Low：信息较弱、主观随口评价，或和产品关系较浅。

## 7. Dashboard Pages

### 7.0 Information Architecture

Dashboard 界面和信息架构默认使用中文。

第一版建议围绕一个核心路径设计：

```text
项目列表 -> 项目概览 -> KOC 频道管理 -> 视频列表 -> 视频分析详情
```

页面设计应优先服务两个动作：

- 快速判断 KOC 到底说了什么。
- 快速复制一版能给产品团队看的文字。

### 7.1 项目列表 / 项目入口

用于查看已有产品项目，并创建新项目。

核心信息：

- 产品/游戏名称。
- KOC 数量。
- 已分析视频数量。
- 最近更新时间。

### 7.2 项目概览

用于查看某个产品/游戏下的视频分析概览。

核心信息：

- 产品/游戏名称。
- 已分析视频数量。
- KOC 数量。
- 高频正向反馈。
- 高频中性观察。
- 高频负向反馈。
- 高频建议。
- 需要产品关注的 Top Issues。
- KOC 列表入口。
- 最近分析视频。
- 项目级可复制产品反馈总结。

项目级可复制产品反馈总结是 MVP 的关键产物，需要汇总当前项目下所有已分析 KOC 和视频的结果。

项目概览页需要提供一段可直接复制给产品团队的文字，建议格式如下：

```text
【KOC 项目反馈摘要】
产品/游戏：
分析范围：当前项目下所有已分析 KOC 视频
KOC 数量：
视频数量：
分析日期：

一、数据范围说明
- 本次报告覆盖 ... 个 KOC、... 条已分析视频。
- 本报告只代表当前项目已分析内容，不代表全网结论。

二、整体结论
- ...

三、正面反馈汇总
1. ...
涉及 KOC：
代表证据：KOC / 视频 / 时间戳 / 原始片段 / 中文翻译

四、中性观察汇总
1. ...
涉及 KOC：
代表证据：KOC / 视频 / 时间戳 / 原始片段 / 中文翻译

五、负面反馈 / 风险汇总
1. ...
涉及 KOC：
代表证据：KOC / 视频 / 时间戳 / 原始片段 / 中文翻译

六、高频问题 / 高频观点
1. ...
出现情况：... 个 KOC / ... 条视频提到
代表证据：KOC / 视频 / 时间戳 / 原始片段 / 中文翻译

七、产品可考虑建议
1. ...
来源类型：KOC 明确建议 / 基于反馈推导 / 需要更多样本验证
依据：来自哪些反馈或证据片段

八、证据索引
1. KOC / 视频 / 时间戳 / 原始片段 / 中文翻译
```

### 7.3 KOC 频道管理

用于管理项目下关注的 KOC YouTube 频道。

核心能力：

- 添加 KOC YouTube 频道链接。
- 展示频道名称和频道链接。
- 开启或关闭自动抓取。
- 选择自动抓取周期：每天、每 2 天、每周。
- 手动触发抓取最近 1 条视频。
- 展示最近抓取时间、下次抓取时间和抓取状态。

### 7.4 视频列表

用于管理和查看已录入的视频。

字段建议：

- 视频标题
- KOC/频道
- 发布时间
- 视频链接
- 分析状态
- 相关片段数量
- 主要倾向
- 高优先级反馈数量

分析状态：

- Pending：已录入，待处理。
- Processing：处理中。
- Done：分析完成。
- Failed：处理失败，需要重试或人工检查。

### 7.5 视频分析详情

用于查看单个视频的详细分析。

核心模块：

- 视频基础信息。
- AI 生成的整体摘要。
- 和产品相关的关键片段列表。
- 有效反馈列表。
- 正面/中性/负面分布。
- 证据片段。

证据片段不单独做独立页面，第一版直接放在视频详情页中。

每条证据至少包含：

- 时间戳。
- 原始字幕片段。
- 原始字幕片段的中文翻译。
- AI 提炼后的含义。
- 反馈倾向：Positive、Neutral、Negative。
- 重要性。
- 视频来源链接。

## 9. Functional Requirements

### 9.1 Project Management

- 用户可以创建一个产品项目。
- 产品项目至少包含产品/游戏名称和产品关键词。
- 产品关键词由用户手动输入，系统提供基础默认关键词供用户增删。
- 用户可以在项目下添加多个 KOC YouTube 频道。
- 同一个项目下的视频分析结果可以被汇总。

### 9.2 KOC Channel Intake

- 用户可以提交 KOC YouTube 频道链接。
- 第一版只支持公开可访问的 YouTube KOC 频道。
- 系统需要解析频道信息，并获取频道名称、频道 ID、频道链接等基础信息。
- 如果频道链接无效，系统需要给出明确错误信息。
- 用户可以在同一个项目下添加多个 KOC 频道。
- 用户可以为每个 KOC 频道选择是否开启自动抓取。
- 如果开启自动抓取，用户需要选择抓取周期：每天、每 2 天、每周。

### 9.3 Video Collection

- 第一版只分析公开可访问的 YouTube 视频。
- 系统需要根据 KOC 频道获取对应视频列表。
- 系统需要识别哪些视频尚未分析，并自动加入分析队列。
- 系统需要获取视频标题、发布时间、视频链接等基础信息。
- 自动抓取开启时，系统在下一个抓取周期自动检查该 KOC 频道，并只分析上次抓取后新发布的视频。
- 自动抓取关闭时，系统不会周期性检查该 KOC 频道。
- 无论自动抓取是否开启，用户都可以手动触发抓取。
- 手动抓取需要立即抓取该 KOC 频道最近 1 条视频，并进入分析流程。

### 9.4 Transcript Handling

- 系统优先使用 YouTube 字幕。
- 如果字幕有多语言版本，优先选择英文或用户指定语言。
- 如果视频没有可用 YouTube 字幕，系统需要进入音频转写流程。
- 字幕或音频转写文本需要按时间片段保存，便于证据追溯。
- 系统需要记录文本来源：YouTube 字幕或音频转写。
- 如果字幕获取和音频转写都失败，系统需要显示失败原因，并允许后续重试。
- 长文本需要切分后再进入 AI 分析，避免上下文过长导致遗漏。

### 9.5 Relevance Detection

- 系统需要根据产品名称、关键词和语义相似度判断片段是否和目标产品相关。
- 只保留和产品、竞品、版本、玩法、用户反馈、社区反应相关的片段。
- 对明显无关的闲聊、广告口播、纯剧情直播过程，应降低优先级或过滤。
- 相关性判断需要给出简短原因，便于人工检查。

### 9.6 Feedback Extraction

- 系统需要从相关片段中提炼有效反馈。
- 每条反馈都必须包含原始证据。
- 多个片段表达同一观点时，应合并为一条反馈，并保留多个证据。
- 系统应区分 KOC 本人的观点和 KOC 转述玩家/社区的观点。

### 9.7 Report Generation

- 系统需要生成一段项目级中文报告，默认面向产品团队。
- 项目级报告需要汇总当前项目下所有已分析 KOC 和视频。
- 项目级报告需要包含数据范围说明、整体结论、正面反馈汇总、中性观察汇总、负面反馈/风险汇总、高频问题/高频观点、产品可考虑建议、证据索引。
- 项目级报告采用正面反馈、中性观察、负面反馈/风险并列的结构。
- 高频问题/高频观点需要体现涉及 KOC 数量或视频数量，帮助产品判断是否为个别意见。
- 产品建议需要标注来源类型：KOC 明确建议、基于反馈推导、需要更多样本验证。
- 报告中的证据需要保留原始片段，并提供中文翻译。
- 报告应使用清晰、可转发的业务表达，不直接堆砌字幕原文。
- 报告中每个关键结论都应能追溯到时间戳。
- 用户可以一键复制报告正文。

## 10. AI Output Requirements

AI 输出必须满足以下约束：

- 不编造没有证据支持的观点。
- 不把 KOC 的疑问句直接当成明确结论。
- 不把游戏实况中的角色行为误判为 KOC 对产品的评价。
- 对低置信度判断要标记为 Low confidence。
- 原始证据必须保留 KOC 的原文表达，并提供中文翻译。
- 建议类内容必须尽量转化成产品可理解的动作，例如“优化新手引导节奏”而不是“感觉这里怪怪的”。

## 11. MVP Workflow

1. 用户创建或选择一个产品项目。
2. 用户录入产品/游戏名称和产品关键词。
3. 用户在项目下添加 KOC YouTube 频道链接。
4. 用户选择是否开启自动抓取；如果开启，需要选择每天、每 2 天或每周。
5. 如果开启自动抓取，系统在下一个周期获取该 KOC 频道上次抓取后新发布的视频，并进入分析流程。
6. 无论是否开启自动抓取，用户都可以手动触发，系统立即抓取最近 1 条视频并进入分析流程。
7. 系统将待分析视频加入分析队列，并优先获取 YouTube 字幕；无字幕时进行音频转写。
8. 系统识别产品相关片段。
9. 系统提炼有效反馈和建议。
10. 系统生成 Dashboard 分析结果。
11. 用户检查证据片段。
12. 用户在项目概览页复制最终项目级产品反馈总结。

## 12. Non-Goals for MVP

第一版暂不做：

- 自动发现用户未添加的 KOC 或全网所有相关视频。
- 本地上传视频分析。
- YouTube 评论区分析。
- 多平台内容整合。
- 复杂 KOC 画像。
- 实时监控和预警。
- 精准舆情趋势预测。
- 自动生成产品 Roadmap。
- 完整权限管理。

这些能力可以作为后续版本扩展，但不应阻塞第一版可演示闭环。

## 13. Success Metrics

MVP 是否成功，主要看以下指标：

- 添加 KOC 频道后，系统能自动获取该频道的视频并产出结构化分析结果。
- 用户不看完整视频，也能理解指定 KOC 对产品说了什么。
- 每条关键结论都能追溯到原始片段和时间戳。
- 项目级最终报告能直接复制给产品团队，不需要大幅重写。
- 人工检查后，核心反馈没有明显遗漏或严重误判。
- Demo 过程中，非技术同事也能理解工具解决的问题。

## 14. Acceptance Criteria

第一版验收标准：

- 用户可以创建产品项目，并输入产品/游戏名称和关键词。
- 用户可以在项目下添加 KOC YouTube 频道链接。
- 用户可以为 KOC 频道开启或关闭自动抓取。
- 自动抓取开启时，用户可以选择每天、每 2 天、每周。
- 自动抓取开启后，系统在下一个周期自动抓取并分析上次抓取后新发布的视频。
- 无论自动抓取是否开启，用户都可以手动触发抓取，系统立即抓取最近 1 条视频并分析。
- 系统能展示 KOC 频道基础信息和对应视频列表。
